# blue-ocean-finder · 缝隙机会发现引擎

## 触发与输入解析

**标准触发格式：`蓝海品类洞察-{方向}`**（连字符后即输入方向）
- `蓝海品类洞察-手机摄影配件` → 方向 = "手机摄影配件"
- `蓝海品类洞察-厨房内容创作装备` → 方向 = "厨房内容创作装备"
- 也接受空格/冒号分隔：`蓝海品类洞察 手机摄影配件`、`蓝海品类洞察：手机摄影配件`

解析规则：取"蓝海品类洞察"后的连字符/空格/冒号之后的全部文字作为**方向输入**，按粒度规则校验（太细/太粗给提示），然后进 playbook。若只说"蓝海品类洞察"不带方向 → 反问用户要一个方向。

## 这是什么 / 和 zach-product-research 的分工

- **blue-ocean-finder（本 skill）= 找方向**：给一个空间 → 扫出"在涨 + 竞争浅"的缝隙候选 shortlist。
- **zach-product-research = 深挖**：对选定的 1 个候选做完整 Go/No-Go + 四件套报告。
- 本 skill 跑到 shortlist + 决策点①，把选中的候选**交接 JSON** 给 zach-product-research。

## 核心原则（必须理解，否则会找错方向）

**真蓝海往往不在一个干净的亚马逊类目节点里——若在，早被那个节点的头部占了。** 所以：
- ❌ 不要按类目树（category_tree）下钻找 spec
- ✅ 要从"**用例 + 搜索需求增速**"自上而下找
- 类目节点是**最后一步**，用来解释白空间根因 + 决定上架去哪，不是发现来源

> 实证：overhead 顶拍 rig 是靠 "overhead tripod +80% YoY" 这个搜索信号先冒出来的；查节点发现 5 个对标品散在 5 个节点跨 2 大类、无节点"拥有"顶拍 = 白空间根因。这是事后解释，不是发现来源。

## 铁律（踩过的坑，必须遵守）

1. **搜索增速先行，禁用销量代替判需求。** P1 的 `keyword_trend` 搜索增速必须先跑；销量是滞后指标、可能是 Q4/促销堆的。massager Back 就是用销量代跑、搜索段 -48% 在跌没发现，白做。需求真假只认搜索趋势。
2. **看竞品销量先拆促销窗口。** 单日/单月销量可能是 Prime Day/Deal 尖峰。判稳态要看多月趋势 + 价格趋势（价格回弹→销量跌=促销依赖）。别拿尖峰当锚（overhead 137单/天是 Prime Day 尖峰，稳态仅 50-70/天且在降）。
3. **搜索池大 ≠ 机会大。** 大搜索词的量常被"别的形态"占据（overhead 节点量被 $16 廉价夹灯吃掉，我方 rig 形态天花板只 2.5-4k/月）。TAM 要按**我方能承接的形态**封顶，不是按搜索池。
4. **发现≠能做。** shortlist 出来只是"缝隙存在"，还要过**可行性闸门**（P4）——IP 雷、启动现金、形态评分天花板会让一个"看起来 GO"的缝隙实际做不了或不赚钱（overhead 就是补了 9 项可行性验证后，GO 7.5 → 6.7）。
5. **不信工具自动凑的数字。** 每个结论核到 Sorftime 原始返回（见 zach 那次 7.0K 假数教训）。
6. **段1❌不等于自动弃——先看有没有"红海质量位移"迹象。** 若头部竞品（含leader）差评集体复现同一组工程缺陷+形态IP干净+稳态销量真实，把"弃"和"质量位移深挖"两条路都摆给用户选，不要替用户拍板（magic arm 案例走的就是这条，段1 0/19通过但质量位移出了CONDITIONAL GO 6.2）。判定细则见 `references/gap_signature.md` 「路径B」。
7. **对标 ASIN 必先查变体结构 + 上架时间。** Sorftime `product_detail`/`product_trend` 的月销是**家族（父体聚合）口径**，亚马逊页面评论数是**家族合并展示**。任何"低评论+高销量"信号，先跑 `product_variations` 拆子体真实销量、查上架日期，排除变体错觉后才能当证据。实证：overhead"24评论撑2k月销=切入成本极低"是把 1,777 评、2023 年上架老链接的家族销量记到了新子体头上（子体真实月销仅 200-300），v3 条件性 GO 因此翻盘（v4 降级 WATCH）。**product_reviews 返回父体评论，本身就是变体家族的信号，见到就要查。**
8. **事实变了，分数必须跟着变。** 每轮补验证/交叉验证后按评分卡机械重算，禁止"风险挂起不入分"。实证：overhead 10_cross 发现三源退潮仍以程序性理由维持 6.7，属锚定偏差；v4 机械重算 6.2+排除项触发，判定反转。
9. **GO 前必过资金闸门，毛利必须用自家 BOM。** 全口径启动现金（首批货按**新进入者年1锚量**而非在位者稳态 + 广告 + Vine + 周转）≤ 可用弹药才能 GO，超出自动降级，"条件性 GO"不能绕过资金排除项。毛利率禁止沿用工具对在位者的估算（overhead 的 65% 是 Sorftime 对 Evershop 的估算，自家 1688 报价从未做过）；带电/易退品类要计退货率。细则见 `references/feasibility_gate.md` A4。
10. **竞争判断以自然位首页普查为主证据；「空白/唯一/无人做」必须全量口径。** `keyword_search_results`（核心词，自然位P1）是买家真实所见，一次调用性价比最高；`product_search` 是相关性排序有噪音，节点报告有碎片化盲区（形态常跨节点散布）。任何"空白"类结论必须基于首页普查或完整 Top100，禁止用精选子样本下"不存在"判断。实证：teleprompter 用 25/100 精选样本推"纯机械空白"，首页普查一次调用就翻出 Glazon/Geimrsy 已占位 $50-60，空白实为价格带缺口。
11. **词级趋势两道守卫（scanner 已内置，人工分析同样遵守）。** ① 噪音门槛：月搜多月为 0 或中位 <1k 的词禁算 YoY（曾对噪音词算出"-28%"假信号）；② 词形迁移：新词垂直起量时必拉词族相邻词（父词/同义词）对照——若此消彼长、总盘平移，是亚马逊搜索入口变化不是新需求（teleprompter 长尾词 0→6,625 与"teleprompter for iphone"塌缩同步发生）。scanner 输出 ❓/🆕 的词一律转人工验证，不进自动排序。
12. **差异化主张必须过"使用流程自洽走查"，痛点弹药必须同价格段。** 把核心卖点放进用户完整使用流程（开箱→安装→完成任务）逐步走一遍，字面不成立的当场杀掉（"零App依赖"的分光镜提词器——手机本身必须跑提词App，纸面即可证伪）；用高价段产品的差评论证低价段产品差异化属跨段推理，须标注折扣（$220 Elgato 的软件差评 ≠ $35 产品的弹药）。

## 输入（一个品类空间方向）

粒度规则：
- ❌ 太细（单产品，如"三脚架"）——没空间可扫
- ✅ 刚好（空间，如"手机直播配件 / 厨房内容创作装备 / 手机摄影配件"）
- ❌ 太粗（如"电子产品 / 3C"）——噪音爆炸
- 方向要贴用户能力圈，否则扫出来也做不了

## 两个决策点（其余全自动）

- **决策点①（选深挖谁）**：shortlist 出来后，让用户选 1 个进深挖。
- **决策点②（最终 GO）**：zach 深挖出 Go/No-Go 后，用户拍板 GO→RFQ。
- 可选**决策点⓪**：批量扫前确认种子词（省 Sorftime 配额）。

## 流程（playbook）

### P0 种子词
从方向拆出 5-10 个种子词（英文，US 站）。优先选**已在 Sorftime 热搜库**的词（具体长尾常 not-hot）。

### P1 关键词宇宙 + 增速扫描（需求侧·段1）
1. 对每个种子用 `keyword_extends` 扩词 → 汇成候选词宇宙（去掉量<1000 的）。
2. 对宇宙里 ~20-40 个词，**逐个用 `keyword_trend` 拉趋势**，原始返回存成 `trends.json`（list）。
3. 跑引擎打分排序：
   ```
   python3 scripts/keyword_growth_scanner.py trends.json
   ```
   引擎输出：每个词的 YoY / 最新月搜 / CPC / **缝隙段1分**(0-10) / 🟢涨⚪平🔴跌 / 数据质量。
   **段1分≥7 且 🟢涨 的 → 进段2。**
   **标 ❓不可评 的词**（月数<13/多月为0/中位<1k）不进自动排序：噪音词直接弃；**标 🆕垂直起量的新词**先做词族迁移检查（铁律#11）——拉父词/同义词 trend 对照，总盘平移则按迁移处理、可当承接词布局但不当"新增需求"证据。
   - 工程现实：Sorftime MCP 只能本 skill 交互调，标准脚本调不了。所以**数据由你(LLM)用 MCP 拉好存 JSON、脚本只打分**。

### P2 竞争验证（供给侧·段2）
对段1幸存词，三件套：
1. **`keyword_search_results`（自然位P1）首页普查——竞争判断的主证据**（铁律#10）：逐条看首页 listing 的形态/价格/销量/品牌，这是买家真实所见。「空白/唯一」结论只能出自这里或完整 Top100。
2. `keyword_detail`（首页评论数分布：<100/<500 占比，越高=竞争越浅）。
3. `product_search`（头部评论数、有无统治者、价格带）——注意它是相关性排序，作辅助不作主证据。
**"低评论+高销量"的样本必须先跑 `product_variations` 查变体 + 上架时间（铁律#7）**——评论壁垒按"买家页面所见（家族合并）"口径判，销量按子体真实口径判。
**缝隙签名 = 段1(需求涨) + 段2(竞争浅、无统治者、价格能撑利润)。** 两段都过 = 候选。

### P3 形态/痛点速览 → shortlist
对候选用 `product_search` 看产品形态分布（找白空间形态）+ 抽 1-2 个头部 `product_reviews` 看痛点是否"可工程解决"。
- **形态天花板**：白空间形态在节点里能承接多少量？（节点头部若是别的形态，我方形态天花板可能远低于节点总量——见铁律#3）
- **痛点语料代表性**：抽的差评是不是目标形态的？（若拿到的是父体/别的形态评论，痛点分布会偏；overhead 用鹅颈语料 → 断裂占比虚高。标注语料来源，必要时双口径）
- **季节性 + 交期**：对核心词 `keyword_trend` 看峰谷月份，倒推 RFQ→到货交期（≈2.5-3.5月）能否赶上下一个峰。
- **差异化主张自洽走查**（铁律#12）：候选的差异化角度若依赖某个卖点主张，把它放进用户完整使用流程走一遍，字面不成立的在这里杀掉，别带进深挖。
产出 **shortlist**（2-4 个候选，每个带：段1分、竞争浅证据（含首页普查）、白空间形态+形态天花板、痛点+语料来源、季节窗口）。

### P4 可行性快筛（feasibility gate）
shortlist 里每个候选，在交给 zach 前先过一遍**可行性硬验证快筛**（见 `references/feasibility_gate.md`）。重点是 IP 快筛（一票否决）+ 启动现金量级 + 形态评分天花板——挡掉"缝隙真但做不了/不赚钱"的候选，避免浪费 zach 一整套四件套。可选加一步**跨平台交叉验证**（A5 TikTok / A5b Reddit探针 / A5c Google Trends / A5d Walmart），只在 A1-A4 都过关的候选上跑，用来降低误判风险+挖营销素材角度，**不是发现工具**（实测证明 TikTok 节奏跟 Amazon 同步非领先；Reddit 对专业器材零件级话题命中密度天生低于1%，先探针再决定要不要建抓取盘；Google Trends 分歧默认更信 Amazon 站内数据，因为统计的人群不同；Walmart 成本最低(2-3次调用)，用于形态分布独立交叉核对——别指望这几项当 P0 哨兵）。

### 决策点① → 交接给 zach-product-research
用户选 1 个（已过 P4 快筛）→ 把数据整理成 zach 的 payload（见 `references/handoff.md`），并**带上 feasibility_gate 里"zach 深挖必须覆盖"的清单**，调用 `/zach-product-research` 做深挖+四件套+Go/No-Go。

### 审计关（决策点② 之前，强制）
任何 GO / CONDITIONAL GO 报告在拍板前，按 `references/audit_checklist.md` 过一遍（10-20 分钟）。两个实测项目（overhead/teleprompter）的翻盘性错误全部是审计翻出来的——变体口径、假YoY、主张不自洽、分数不一致——把这步制度化比事后返工便宜得多。审计不过 → 修正报告重打分，再进决策点②。

### 决策点② → RFQ
zach 出 Go(≥7.5) → 供应商 RFQ。

## 评分卡（缝隙签名）

见 `references/gap_signature.md`。核心两段：**需求涨(段1) + 竞争浅(段2)**，其余(痛点可解/利润/能力圈贴近)是加分。

## 反模式（别犯）
- 别按类目树下钻找 spec（会漏掉跨节点的缝隙）
- 别只看搜索量大小当"拥挤度"（量是需求规模，拥挤看竞争）
- 别把搜索池大小当形态天花板（大词量常被别的形态占据，铁律#3）
- 别用销量代替搜索判需求（铁律#1）；别拿促销尖峰当稳态锚（铁律#2）
- 别把"低评论+高销量"直接当低壁垒——先查变体结构和上架时间（铁律#7，overhead 变体错觉教训）
- 别跳过可行性闸门就交接 zach（IP/现金/形态评分会翻盘，铁律#4）
- 别信工具自动凑的数字——每个结论核到 Sorftime 原始返回（见 zach 那次 7.0K 假数教训）
- 别用精选子样本下"空白/唯一"结论——全量口径（首页普查/完整Top100）才有资格说"不存在"（铁律#10）
- 别对噪音词算 YoY、别把新词垂直起量直接当新需求（铁律#11，scanner 会标 ❓/🆕）
- 别让卖点主张跳过使用流程走查——"零App"这种洞纸面就能杀（铁律#12）
- 别让旧版结论存活——vN+1 定稿时旧版标 superseded、RESUME/种子池同步（见 handoff.md 交付纪律；overhead v4 出来后 RESUME 还写着 GO 7.5 的教训）
- 同时深挖多个 = 浪费；shortlist 后只深挖 1 个
