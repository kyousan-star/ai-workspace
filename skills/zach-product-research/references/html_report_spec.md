# HTML 精简报告规范

## 必选区块

1. Header
2. **需求漏斗：为什么是这个结论**（`ch01.decision_funnel`，⛔ 必填 ≥4步）——写给第一次看报告、没参与调研过程的人，说明从原始输入/假设到最终结论中间经过了哪些证伪、转折、验证关卡。不加这段，冷读者只看到"CONDITIONAL GO 6.2"这类结论会问"why this"。**不要用第三方工具/人名代称这个阶段**，用描述性说法（如"定向验证阶段""建模量化阶段"）
3. Executive Summary（核心结论卡）
4. KPI 卡片
5. 多维分布表
6. 消费者痛点卡片
7. 供需缺口/机会空白
8. 品牌/竞品格局
9. 进入壁垒评估
10. Go / No-Go评分卡
11. 策略建议
12. Footer

### 需求漏斗写法要点

- 每步标 `stage`：`start`（起点假设）→ `turn`（转折/需要人判断的分岔点，可 ≥1 个）→ `gate`（量化验证关卡，可 ≥1 个）→ `final`（落地结论），首末两端固定。
- 每步给出：这一步在说什么（desc）+ 具体数据支撑（data_points，2-4 条足够）+ 为什么这一步重要/筛掉了什么（why）。
- 目的是让读者能重建"为什么不是别的结论"，不是重复 Executive Summary 的内容。

## 样式约束

- 使用 CSS 变量统一颜色
- 容器 `max-width: 960px`
- KPI 用网格布局
- 表格可读、可响应式缩放
- Footer 必须写清数据来源、采集日期、MD 与 XLSX 文件名

## 常见问题

- 不要只给“一段话 + 一个表格”的简版
- 不要出现“约”“少量”“大概”这类模糊词替代数字
- HTML 与 Markdown / Excel 必须同源，不允许三份文件数字对不上
